# AGENTS.md

## 项目概览

这是一个基于以下技术栈构建的 Web 应用：

- Next.js
- React
- Tailwind CSS
- shadcn/ui
- Supabase

在本仓库中工作时，默认目标如下：

- 优先做最小范围、低风险、局部性的修改
- 优先复用现有组件、样式模式与目录结构
- 保持当前项目已有的命名方式与代码风格
- 在处理 UI 或小功能任务时，避免顺手做无关的大重构
- 遇到多种实现方式时，优先选择与现有项目最一致的方案，而不是“更通用但更重”的方案

---

## 通用工作方式

处理任务时，默认按以下顺序理解上下文：

1. 先阅读当前文件
2. 如有需要，再阅读当前文件直接 import 的本地文件
3. 如有需要，再阅读同目录或同模块下的兄弟文件
4. 只有当任务明确需要时，才继续向更大范围扩展

除非明确要求，否则不要一开始就扫描整个仓库。

对于小型 UI 任务，优先关注：

- 当前页面或当前组件
- 当前页面直接依赖的子组件
- 相关的 hooks
- 相关的 Supabase 查询或数据写入
- 附近的工具函数或通用常量

优先逐步建立局部理解，而不是一次性广泛扫描。

---

## 项目结构理解原则

本项目通常可能包含以下目录：

- `src/app/` 或 `app/`：Next.js 路由入口、layout、page、route handlers
- `src/components/` 或 `components/`：可复用组件
- `src/components/ui/`：shadcn/ui 基础组件
- `src/lib/`：通用工具、辅助函数、共享实例
- `src/hooks/`：自定义 React hooks
- `src/app/api/`：接口路由
- `src/features/`：按业务域拆分的功能模块（如果存在）
- `src/types/`：共享类型定义
- `supabase/`：Supabase 配置、迁移、策略、种子数据、本地开发相关内容（如果存在）

在解释代码或做修改时，尽量识别并说明以下层次：

- 入口文件
- UI 展示层
- 状态与数据处理层
- 服务端交互层
- 可复用公共组件层

## UI 修改规则

处理 UI 任务时，优先顺序如下：

1. 优先复用现有组件
2. 优先复用现有 Tailwind 写法和附近页面的样式模式
3. 优先复用现有 shadcn/ui 组件组合方式
4. 只有在无法合理复用时，才新增局部组件

新增 UI 时应遵循：

- 尽量匹配当前页面附近已有的间距、字号、圆角、边框、阴影、布局密度
- 优先沿用当前模块已有视觉结构，不要无故重新设计页面
- JSX 层级保持清晰，避免为小改动引入过度嵌套
- 小型 UI 增加优先放在当前文件或当前模块内解决，不要为了小功能抽出过度通用的抽象

除非明确要求，不要顺手重做无关区域的设计。

---

## React / Next.js 规则

### 组件修改

- 优先保持现有组件职责边界
- 不要为了小修改拆分出过多新组件
- 仅在以下情况下考虑抽组件：
  - 当前 JSX 已明显过长且重复
  - 存在明确复用需求
  - 抽离后能明显提升可读性

### Client / Server 边界

- 注意 Next.js 中 Server Component 与 Client Component 的边界
- 不要无必要地把组件改成 `"use client"`
- 如果任务仅涉及静态 UI 或服务端可完成的数据读取，优先保持服务端方案
- 只有在需要浏览器交互、状态、事件、ref、effect 等能力时，才使用 Client Component

### 路由与页面

- 修改页面时，优先理解当前 page、layout、子组件之间的关系
- 不要轻易调整路由结构、文件位置、layout 层级，除非任务明确要求

---

## Tailwind CSS 规则

- 优先沿用当前文件或相邻组件中已经出现过的 Tailwind class 组合
- 不要仅仅因为“更优雅”就大面积改写已有 class
- 小型修改时，尽量只调整必要 class
- 对于重复出现的复杂样式，只有在确认有复用价值时再考虑抽象

处理 Tailwind 时应注意：

- 保持响应式写法风格与现有代码一致
- 保持间距与字号的相对关系一致
- 尽量避免引入与现有页面视觉节奏冲突的新模式
- 如果项目已使用设计 token 或 CSS 变量，优先使用现有变量体系

---

## shadcn/ui 使用规则

- 优先使用项目中已经在用的 shadcn/ui 组件
- 如果已有封装组件，优先使用封装层，而不是直接重新拼原始 ui 组件
- 不要无必要新增新的基础 UI 封装方式
- 保持 variant、size、结构风格与现有使用方式一致

如果要新增基于 shadcn/ui 的组合式 UI：
- 先参考相邻页面已有写法
- 保持命名、结构、样式层次一致
- 避免为了一个页面单独发明完全不同的交互模式

---

## 数据与 Supabase 规则

处理 Supabase 相关代码时：

- 先识别当前数据是在客户端获取，还是在服务端获取
- 先沿用当前模块已有的数据访问方式
- 不要在同一模块里混入风格完全不同的数据获取模式，除非明确要求重构
- 修改查询或写入时，优先保持字段命名和现有数据结构一致

如果任务涉及数据库读写：
- 优先做最小修改
- 不要猜测不存在的字段
- 不要假设数据库约束、RLS 策略、触发器一定存在，除非代码中已经体现
- 遇到不确定的数据结构时，先根据现有代码推断，再明确说明假设

如果任务涉及 upsert / insert / update / delete：
- 注意主键、外键、关联 ID 的来源
- 注意前端传值是否为字符串、数字、空字符串或 null
- 注意列表型数据是否需要先删除旧记录再插入新记录
- 注意返回值是否为数组、单对象或为空

## 修改范围控制

默认情况下：

- 只修改完成任务所必需的文件
- 不顺手修 unrelated 的 lint、格式、命名、结构问题
- 不做无关的大面积重排
- 不重命名文件、目录、导出名，除非任务明确要求
- 不引入新的依赖，除非现有项目无法完成任务且任务明确需要

如果发现“顺手重构”会让改动范围明显扩大，应优先放弃重构，先完成目标任务。

除非明确要求，否则不要添加大量解释性注释。

如果需要写注释：
- 注释应说明“为什么这样做”，而不是重复代码表面含义

## 排查问题时的默认方式

遇到 bug、报错或行为异常时，优先按以下顺序排查：

1. 当前文件自身逻辑
2. props / state / hooks 数据流
3. 直接依赖的子组件
4. API 或 Supabase 数据结构
5. Next.js 的 client/server 边界问题
6. 样式覆盖或布局问题

除非任务明确要求，否则不要一开始就做全仓库搜索式排查。

## 对“熟悉项目”类任务的处理方式

如果任务不只是改代码，还要求逐步理解项目，请按以下方式执行：

- 先从当前页面或当前模块开始理解
- 逐步扩展到直接依赖和兄弟模块
- 用简短方式总结：
  - 当前模块是做什么的
  - 入口文件是什么
  - 关键组件有哪些
  - 数据从哪里来、往哪里去
  - 后续继续阅读建议先看哪些文件

不要一开始就给出空泛的全仓总结，应先给局部、具体、可操作的理解结果。

## 新增组件或函数时的原则

只有在以下情况才新增组件、hook、工具函数：

- 当前文件复杂度明显增加
- 存在清晰复用需求
- 提取后更容易理解，不会增加阅读跳转成本

新增时应遵循：

- 命名与当前模块一致
- 放在最合理、最近的目录
- 不为了“理论上的复用”过早抽象
- 不引入与项目现有模式冲突的新组织方式

---

## 验证与检查

完成修改后，优先做与任务相关的最小验证。

例如：

- UI 改动：优先检查页面结构是否自洽、样式是否与附近一致
- 表单改动：优先检查默认值、提交值、类型转换
- 数据改动：优先检查字段映射、空值处理、返回值结构
- 条件渲染改动：优先检查空态、加载态、边界情况

除非任务明确要求，否则不要运行耗时很重的全量检查。

## 默认决策倾向

当存在多种可行方案时，优先选择：

- 改动更小的
- 风险更低的
- 更贴近当前项目已有模式的
- 更容易让后续维护者理解的
- 更适合当前模块局部演进的

而不是优先选择最“通用”、最“完整”或最“理论正确”的方案。
